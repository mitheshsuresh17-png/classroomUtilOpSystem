Department                                                 ABSTRACT

The Classroom & Lab Utilization Optimization System is designed to manage and analyze classroom and laboratory usage in academic institutions. Manual scheduling methods often lead to inefficient room utilization, double bookings, and scheduling conflicts. This project proposes a centralized database-driven solution to store room details, time slots, departments, courses, batches, and schedules.
The system uses ER modeling to design the database and relational schemas are derived from it. Tables are created using SQL DDL commands and populated using DML statements. Complex SQL queries are written using joins, subqueries, aggregate functions, constraints, views, triggers, and cursors. Normalization is applied up to 5NF to remove redundancy and anomalies. Transaction control, concurrency control, and recovery mechanisms are implemented using TCL and locking commands.
This system helps institutions optimize infrastructure usage and avoid scheduling conflicts.

1.1 Introduction

The Classroom & Lab Utilization Optimization System (CLUS) is a structured database-driven solution designed to modernize and streamline the scheduling and allocation of classrooms and laboratory resources in academic institutions. In large campuses where multiple departments, batches, and courses compete for limited physical infrastructure, manual scheduling methods often result in inefficient room utilization, idle facilities, double bookings, and timetable conflicts. In today's data-centric academic environment, effective infrastructure management depends on accurate, centralized, and well-structured scheduling data. This project moves beyond simple timetable recording to build an integrated relational database environment that supports the complete allocation lifecycle - from room and lab registration to time-slot mapping, batch scheduling, and conflict detection.

At its core, the system addresses the absence of centralized tracking and visibility that commonly exists in manual or fragmented scheduling processes. By implementing a well-designed and normalized relational database model, the project ensures that every scheduling component - including room details, capacity, time slots, departments, batches, and course schedules - is logically organized and interlinked. This reduces redundancy, prevents allocation clashes, and improves transparency in room usage patterns. The resulting database framework enables institutions to analyze utilization, detect conflicts automatically, and support efficient, conflict-free classroom and lab allocation through reliable data management.

1.4 Problem Statement

Manual classroom scheduling in academic institutions often leads to inefficient utilization of rooms, where some classrooms remain idle while others are overbooked. This lack of centralized tracking results in scheduling conflicts and poor visibility of room usage. The proposed system aims to design a database that records classroom details, time slots, and course schedules to analyze utilization and support conflict-free and efficient room allocation.

? Functional Requirements
The system must store complete details of classrooms and laboratories, including room ID, type, and seating capacity.
The system must maintain department information and link it with courses and batches.
The system must record course details such as course ID, course name, and associated department.

The design must be scalable to support additional modules like faculty allocation and exam scheduling.

The database must be secure and reliable, with proper transaction control (commit/rollback).

2.3 Creation of Database and Tables - DDL Commands 

       1) Database Creation:
CREATE DATABASE classroom_utilization_db;
USE classroom_utilization_db;

2) Table Creation:
CREATE TABLE Department (
    dept_id INT PRIMARY KEY,
    dept_name VARCHAR(50) NOT NULL
);

CREATE TABLE Course (
    course_id INT PRIMARY KEY,
    course_name VARCHAR(50) NOT NULL,
    dept_id INT,
    FOREIGN KEY (dept_id) REFERENCES Department(dept_id)
);

CREATE TABLE Batch (
    batch_id INT PRIMARY KEY,
    year_of_study INT NOT NULL,
    section CHAR(1) NOT NULL,
    student_count INT NOT NULL,
    dept_id INT,
    FOREIGN KEY (dept_id) REFERENCES Department(dept_id)
);

CREATE TABLE Room (
    room_id INT PRIMARY KEY,
    room_type VARCHAR(20),
    capacity INT NOT NULL
);

CREATE TABLE Time_Slot (
    slot_id INT PRIMARY KEY,
    day VARCHAR(10),
    start_time TIME,
    end_time TIME
);

CREATE TABLE Course_Schedule (
    schedule_id INT PRIMARY KEY,
    course_id INT,
    batch_id INT,
    room_id INT,
    slot_id INT,
    FOREIGN KEY (course_id) REFERENCES Course(course_id),
    FOREIGN KEY (batch_id) REFERENCES Batch(batch_id),
    FOREIGN KEY (room_id) REFERENCES Room(room_id),
    FOREIGN KEY (slot_id) REFERENCES Time_Slot(slot_id)
);

       4) Utilization Query
SELECT room_id, COUNT(*)
FROM Course_Schedule
GROUP BY room_id;
This shows how many times each room is used.


       5) Find Free Rooms
SELECT room_id
FROM Room
WHERE room_id NOT IN
(SELECT room_id FROM Course_Schedule);
This detects idle infrastructure - main project goal.

6) Conflict Detection Query
SELECT room_id, slot_id, COUNT(*)
FROM Course_Schedule
GROUP BY room_id, slot_id
HAVING COUNT(*) > 1;
This detects double booking conflicts.
